## Process Documents

### For Contributors

Start here if you want to contribute:

1. Read [CONTRIBUTING.md](CONTRIBUTING.md)
2. Check [QUALITY.md](QUALITY.md) for Definition of Done
3. For major features, follow [RFC_PROCESS.md](RFC_PROCESS.md)
4. For architecture, follow [ADR_PROCESS.md](ADR_PROCESS.md)

### For Maintainers

Start here if you're maintaining the project:

1. Read [GOVERNANCE.md](GOVERNANCE.md)
2. Review [QUALITY.md](QUALITY.md) before approving PRs
3. Reference [RELEASES.md](RELEASES.md) when releasing
4. Use [ADR_PROCESS.md](ADR_PROCESS.md) for architectural decisions

### For Decision-Making

When you need to make a project decision:

- Small decision? → See GOVERNANCE.md (Decision-Making: Small)
- Feature proposal? → Open RFC (RFC_PROCESS.md)
- Architecture change? → Create ADR (ADR_PROCESS.md)
- Breaking change? → See RELEASES.md (Breaking Changes Policy)
- Governance question? → See CONSTITUTION.md or GOVERNANCE.md

## Philosophy

Orbit governance is built on these principles:

✅ **Transparent** — All decisions public; no secret deliberation  
✅ **Principled** — Guided by CONSTITUTION.md, not politics  
✅ **Scalable** — Works from 1 maintainer to 100+  
✅ **Modular** — Documents can evolve independently  
✅ **Stable** — Constitution frozen; governance processes flexible  

## Key Principles

**The Golden Rule**  
Orbit exists to make Docker Compose deployments safer—not more complicated.

**Documentation Constitution**  
A feature is not complete until its documentation is complete.

**Definition of Done**  
Features must meet quality standards; this is non-negotiable.

**Architectural Boundaries**  
Docker Engine, Compose, and Rollout each own specific responsibilities.
