# Orbit — Backend Identity and Ownership Truth Review

**Date:** 2026-07-15
**Scope:** Determine the single authoritative backend identity/ownership model (or the absence of one) behind the pre-market audit's two P0 findings. No code was modified. No fix was implemented.
**Method:** Direct source reading only — `internal/compose/generator.go`, `internal/proxy/{reconciler,recovery,registry,project_registry,eventsource,health_controller,project_health_controller}.go`, `internal/rollout/rollout.go`, `internal/api/control.go`, `internal/state/{state,recovery,invariants}.go`, `cmd/docker-orbit/main.go`. Every conclusion below cites exact file/function and is tagged **code-proven**, **test-proven**, **live-proven**, **inferred**, or **unverified**. Nothing here is taken from the prior audit's own conclusions without independent re-verification against current source.

---

## 1. Executive Finding

**The two P0 findings are not independent bugs. They are two different visible symptoms of one missing thing: Orbit has never defined a single, explicit backend-identity-and-ownership contract that every Docker-truth-reading component is required to implement. Instead, three components each built their own partial, mutually-incompatible answer, at three different points in the project's history, and nothing written down says which one is authoritative.**

Concretely:
- `internal/proxy/recovery.go`'s `DockerRecoverySource` (built first) has the *most* complete answer: an explicit instance-ownership check, a static-label extraction path, and a separate, dynamic-ID-aware direct-verification path (`VerifyBackendByID`) that already documents, in its own comment, the exact two-namespace problem the audit's Finding 1 rediscovered independently.
- `internal/proxy/reconciler.go`'s `Reconciler` (built later, ADR-0006 Stage 4) explicitly and deliberately omits the instance-ownership check — citing an ADR-0006 claim ("ProjectRegistry's native per-service keying supersedes it") that is **correct for cross-service isolation within one process** and **silently wrong for cross-project isolation between processes**, a distinction ADR-0006's own text does not draw. It also never gained an equivalent of `VerifyBackendByID`'s dynamic-ID awareness.
- `internal/rollout/rollout.go`'s own container discovery (`docker ps --filter label=com.docker.compose.service=<service>`) filters on a *different* label vocabulary again — Docker Compose's own standard label, not Orbit's — and, like Reconciler, applies no project scope.

All three problems — the rollout/reconciler backend-ID collision, the cross-project traffic leak, and the previously-flagged "rollout not scoped to a Compose project" bug — trace to the **same missing ingredient**: nothing anywhere in the discovery/matching layer asks "does this container belong to *my* deployment," using the one label Docker Compose already attaches for free (`com.docker.compose.project`) and that Orbit has never once read. **[code-proven — confirmed by exhaustive grep, zero references to `com.docker.compose.project` anywhere in the repository.]**

## Final Decision: **B — Missing Identity Invariant**

A new ADR is required before implementation. This is not "one component has a bug" (option A) — three independently-evolved components each answer a question ("what identifies/owns this backend?") that was never given one authoritative answer, and the fix touches label emission (generator), two discovery paths (Reconciler, rollout's compose calls), and one lookup convention (the hardcoded mesh network name) in a coordinated way. It is also not "an architecture/ownership review" in the sense of moving package boundaries (option C) — every fix identified below fits inside the existing package structure and existing invariants (Docker is the sole source of truth; Registry does no I/O; Reconciler is additive, not a second recovery engine). See §12 for the precise scope of what the ADR must define.

## 3. Backend-ID Call-Site Matrix

| Action | File:Function | What this code believes a backend ID represents |
|---|---|---|
| **Creates** (static) | `internal/compose/generator.go`, `buildBackingService` (`backing.Environment["ORBIT_BACKEND_ID"] = name + "-default"`) | "The one backend this service has, before anything has ever changed" — a per-service constant, not a per-container value |
| **Creates** (dynamic) | `internal/rollout/rollout.go:396`, `Run` (`newBackendID := opts.Service + "-" + shortID(newID)`) | "This one specific, real, running container, distinguishable from any other replica of the same service" |
| **Reads** (static, no ownership check) | `internal/proxy/reconciler.go:261-303`, `extractBackend` | "Whatever `ORBIT_BACKEND_ID` says is authoritative for this container" — believes static labels are sufficient identity, full stop |
| **Reads** (static, with ownership check) | `internal/proxy/recovery.go:226-294`, `extractBackend` | Same static belief, **plus** "but only if the container's `orbit.io/proxy-instance` also matches who I am" |
| **Reads** (dynamic, ID-first) | `internal/proxy/recovery.go:315-371`, `VerifyBackendByID` | "The ID itself encodes a real container ID — go inspect that container directly, don't trust labels at all for this case" — the only place in the codebase that treats the dynamic scheme as first-class |
| **Compares/collides** | `internal/proxy/reconciler.go:206-219`, `reconcileService` | Detects two containers producing the same static ID and logs it as a "collision" to be resolved by iteration order — treats this as an anomaly to tolerate, not a signal that its own identity model is incomplete |
| **Persists** | `internal/rollout/rollout.go` → `MarkTransitioning`/`CommitAuthority` → `internal/api/authority.go` → `internal/state/state.go` (`ActiveGenerationState.ActiveGeneration`, `RolloutState.{Old,New}Generation`) | Stores the dynamic ID under the field name "generation" — conflating two concepts textually, though the stored value is the correct, unique, dynamic one |
| **Logs** | Throughout `reconciler.go`, `recovery.go`, `rollout.go`, `control.go` — consistently logs whichever ID string the local code path constructed, no cross-referencing | Each component logs its own belief without ever cross-checking it against the other scheme |
| **Removes** | `internal/proxy/registry.go:172-181`, `Remove(id string)` | Pure string match — no semantic check at all; whichever caller asks for removal by a given string succeeds if that string happens to be a current key |
| **Accepts externally, unvalidated semantically** | `internal/api/control.go:504-553`, `addBackend` → `validateBackendID` (`control.go:492-502`) | Only validates length and absence of control characters — **the control API has no opinion on what a valid backend ID means**, it is pure pass-through to `Registry.Add` |

**Conclusion: the current code does not have one backend identity model. It has (at least) three: a static-label model (Reconciler, and half of Recovery), a dynamic-ID model (Rollout, and the other half of Recovery via `VerifyBackendByID`), and a "whatever string the API caller says" model (the Registry/control API layer, which is identity-agnostic by design and correctly so — it is not this layer's job to enforce semantics, but nothing above it currently guarantees semantics either). [code-proven]**

## 4. Rollout Identity Model

**Constructs**: `<service>-<12-char-container-id>` (`rollout.go:396`), directly from the real Docker container ID it just created via `docker compose up --scale`. This ID is:
- Never written to any Docker label or env var on the container itself — it exists only in the rollout CLI process's own memory, the control API's `POST /backends` call, the proxy's in-memory `Registry`, and persisted rollout/authority state (`internal/state`).
- Guaranteed unique per container by construction (it embeds the real container ID).
- **Not derivable from Docker alone after a crash** — this is precisely why `VerifyBackendByID` exists: it recovers the model by parsing the container-ID substring back out of the ID string and doing a direct `ContainerInspect`, rather than by scanning labels.

Rollout's own container-*discovery* (as opposed to backend-ID construction) is a **separate, third mechanism** again: `findOldContainer`/`serviceReplicaCount` (`rollout.go:791,831,968,1114`) use `docker ps --filter label=com.docker.compose.service=<service>` — Docker Compose's own standard label, not any Orbit-specific one, and with **no project scope** (`com.docker.compose.project` is never added to the filter). This is the mechanism behind the previously-identified "rollout creates orphaned shadow projects" bug, and it shares the identical missing ingredient as Finding 6. **[code-proven, `rollout.go:791,831,968,1114`]**

## 5. Reconciler Identity Model

**Constructs**: nothing — Reconciler never assigns an ID, it only *extracts* one, purely from the static `ORBIT_BACKEND_ID` env var (`reconciler.go:269-278`). It has:
- No instance-ownership check (explicitly omitted, `reconciler.go:253-260`'s own doc comment cites ADR-0006).
- No project scope in its `ContainerList` filter (`reconciler.go:126-129`: `label=orbit.io/managed=true, status=running` only).
- No dynamic-ID awareness — no equivalent of `VerifyBackendByID` exists in this file or is called from it.
- No parameter through which it could even perform an instance check — `NewReconciler` (`reconciler.go:63`) takes `(pr *ProjectRegistry, docker containerLister, m ReconcilerMetrics, log *zap.Logger)`, no proxy-instance/project identifier at all, confirmed by its call site (`main.go:541`: `proxy.NewReconciler(pr, reconcilerDocker, m, log)`) — compare to `NewDockerRecoverySourceWithConfig(service, log, ...)` (`main.go:788,796`), which **does** take the per-service instance identifier. **This is a constructor-level omission, not just a missing `if` inside a function body — Reconciler was never wired with the information it would need to perform the check recovery.go already does.** **[code-proven]**

Reconciler's model believes: "the set of Docker-observable, statically-labeled facts about a container is a complete and sufficient identity" — an assumption that is false the instant two replicas of one service (the ordinary, intended state during every rollout's transition window) exist simultaneously, and false again the instant two different Compose projects use the same service name.

## 6. Recovery Identity Model

`DockerRecoverySource` (`recovery.go`) has the most complete model of the three, built in two layers:
1. **Label-based, ownership-checked scan** (`extractBackend`, `recovery.go:226-294`): requires `service`, `proxy`, `generation` labels all non-empty, **and** requires `orbit.io/proxy-instance` to match its own `d.proxyInstance` field (set per-service at construction time, `main.go:788,796`) — this check is correctly scoped *within one process*, because `DockerRecoverySource` is instantiated once per service, with that service's own name as `proxyInstance`, in both legacy and shared-proxy topologies alike (`executeRecoveryForProject`'s per-service loop, `main.go:~760-820`). **This means Recovery's ownership check does not have the cross-service confusion problem Reconciler would have if two of the process's own services collided — but it has exactly the same gap as Reconciler for the cross-*project* case, because `orbit.io/proxy-instance`'s value is just the service name, and two different projects using the same service name produce the same value.** **[code-proven]**
2. **Dynamic-ID-based direct verification** (`VerifyBackendByID`, `recovery.go:315-371`): the only code in the entire repository that treats the rollout's dynamic ID scheme as first-class. It explicitly documents (comment, `recovery.go:296-314`) the exact two-namespace problem underlying Finding 1, and works around it correctly for its own purpose (startup/on-demand recovery) by parsing the container-ID suffix out of the ID string and inspecting that container directly — never trusting the static label for this case. Its own ownership check (`recovery.go:337-343`) re-derives the same instance-match logic as `extractBackend`.

**Recovery's model is the closest thing the codebase has to a correct answer — but it is scoped to "this one process's own services," it is not exposed as a reusable, documented invariant other components are required to adopt, and it still shares Reconciler's blind spot for the cross-project case.** **[code-proven]**

## 7. Ownership-Proof Comparison

| Component | Proof used | Cross-service (within one process) | Cross-project (same host) |
|---|---|---|---|
| Rollout | None at the identity level — trusts its own dynamic ID construction and Docker's `--filter label=com.docker.compose.service=<service>` for discovery | N/A (operates on one named service at a time) | **No proof — filter has no project scope** |
| Reconciler | None | Protected only incidentally, by `ProjectRegistry`'s per-service map keys (i.e., protected by *data structure*, not by verifying the container's claim) | **No proof at all** |
| Recovery (`DockerRecoverySource.extractBackend`) | `orbit.io/proxy-instance` label == own `d.proxyInstance` | Protected explicitly and correctly | **No proof — the label's value (service name) is not project-unique** |
| Recovery (`VerifyBackendByID`) | Same instance-match check, plus a direct container-ID-derived `ContainerInspect` | Protected | **No proof — same limitation** |
| EventSource | Passes Docker event payloads through to trigger reconciliation; never itself constructs or verifies identity (`internal/proxy/eventsource.go` — confirmed no label/ownership logic exists in this file at all) | N/A — delegates entirely to whichever of the above it triggers | N/A — same |
| HealthController / ProjectHealthController | Operates purely on already-registered `Registry` entries by ID and address — never re-derives identity from Docker at all | N/A | N/A |

**No two components use the same ownership proof, and none of them are project-scoped.** The closest thing to a host-wide uniqueness guarantee anywhere in this table is Docker Compose's own automatically-applied `com.docker.compose.project` label — which literally zero lines of Orbit code read. **[code-proven, exhaustive grep confirms zero references]**

## 8. Compose-Project Isolation Analysis

Isolation *within* one proxy process, across that process's own configured services, is genuinely solid: `ProjectRegistry` (`project_registry.go`) gives each service its own `*Registry` instance with its own lock, and `requireProvableService` (`control.go`) prevents any mutating control-API request from resolving to the wrong service without naming it explicitly. This was independently confirmed live in the pre-market audit (two services, continuous dual traffic through a real rollout, zero cross-service leakage).

Isolation *between* independent Orbit deployments (different Compose projects, potentially different teams, on the same Docker host) **does not exist as a designed property at all** — it is an accidental side effect of operators choosing non-colliding service names, which nothing in the codebase requires, validates, or warns about. The mechanism is: (a) `internal/compose/generator.go`'s `buildBackingService` labels every backing container with `orbit.io/service = <raw compose service name>`, with no project prefix or suffix ever added; (b) `Reconciler.ReconcileOnce`'s `ContainerList` filter (`reconciler.go:126-129`) and `DockerRecoverySource`'s equivalent (`recovery.go:73-77,116-119`) both query **every** `orbit.io/managed=true` container on the host, unfiltered by project; (c) the grouping/matching step that follows uses only the (non-project-scoped) service-name label to decide "does this container belong to me." **[code-proven, live-proven independently by 5 of 6 pre-market audit agents]**

## 9. Global Mesh-Network Dependency Analysis

**Why `docker_rollout_mesh` is hardcoded globally**: the *only* stated reason anywhere in the codebase is `internal/compose/generator.go:52-57`'s own comment: without pinning the literal name, Compose would prefix it per-project (e.g. `myproject_docker_rollout_mesh`), and `internal/proxy/recovery.go`'s **exact-match** lookup `inspect.NetworkSettings.Networks["docker_rollout_mesh"]` (`recovery.go:267,347`) would never find the container's IP. `internal/proxy/reconciler.go:281` independently duplicates the identical exact-match pattern (its own doc comment, `reconciler.go:253-256`, states it "mirrors `DockerRecoverySource.extractBackend`'s label/env parsing").

**Is this part of Orbit's architecture, or a workaround for another lookup assumption? — A workaround, decisively.** Direct counter-evidence exists in the same codebase: `internal/rollout/rollout.go:1062-1084`'s `pickMeshIP` already solves the identical problem — selecting a container's mesh IP out of a set of `"networkname=ip"` tokens — using `strings.HasSuffix(name, "docker_rollout_mesh")`, which works correctly regardless of Compose's per-project prefix. **Rollout's own code does not need the network to be globally named; only Reconciler's and Recovery's exact-match map lookups do, and only because they were written as a direct map-key access (`m["docker_rollout_mesh"]`) instead of an iterate-and-match-by-suffix, the exact pattern already proven to work elsewhere in this same repository.** **[code-proven — `rollout.go:1079` vs. `recovery.go:267,347` and `reconciler.go:281`, directly contradictory conventions for the identical problem within one codebase]**

This means the global network name is not load-bearing for Orbit's actual routing architecture (proxy-owns-the-port, TCP passthrough) — it is load-bearing *only* for two specific lines of lookup code that could be rewritten to match the pattern `rollout.go` already uses. **Do not rename the network without first fixing those two lookups to suffix-match — but fixing those two lookups removes the only reason the global name exists.** Separately and importantly: **even after fixing the network-name workaround, this alone does not close the cross-project discovery leak (§8)** — `ContainerList`'s label filter is queried *before* any network-membership check ever happens, so an unrelated project's container can still be matched by service-name label collision even if it's on a different, correctly-project-scoped network. The network-name fix and the discovery-scoping fix are two separate, complementary changes, not one.

## 10. Identity Disagreement Root Cause

Restated precisely: Orbit was built in at least three distinct phases (pre-existing single-service proxy → ADR-0006 shared-proxy/registry work → this session's own audit), and each phase's new component (`recovery.go` first, then `reconciler.go` for ADR-0006 Stage 4, then `rollout.go`'s dynamic-ID scheme, which actually predates both but was never reconciled with either) solved "how do I know this container is mine" using whatever Docker-observable facts were convenient at the time, without a written contract obligating agreement. `recovery.go` got the closest to a correct, complete answer (instance-check + dual static/dynamic paths) specifically because its author had to solve the "label scan can't find a per-rollout ID" problem directly and documented the reasoning thoroughly (`recovery.go:296-314`). `reconciler.go`'s own doc comment then explicitly, consciously chose to drop half of that solution (the instance check), reasoning correctly about cross-service safety but never examining cross-project safety — a reasoning gap in ADR-0006 itself, not an implementation slip.

## 11. Required Production Invariant

The existing invariant — "Docker is the only source of truth" — is necessary but **not sufficient**, and its insufficiency is precisely what allowed this bug class to exist while that invariant was never violated in the narrow sense (nothing here trusts stale persisted state over live Docker; §12 of the pre-market audit already confirmed this holds). The missing invariant, which this review recommends stating explicitly:

> **Every component that discovers, matches, verifies, or claims ownership of a Docker container as an Orbit backend must derive both (a) that backend's identity and (b) the proof that this specific proxy process is entitled to act on it, from the same complete set of Docker-observable facts — and that set must include a project/deployment scope, not service name alone.**

Concretely, this requires (not implemented here, this is a definition, not a patch): (1) one agreed backend-identity representation that is either always static-derivable or always includes a documented, universal fallback to dynamic direct-verification — not two representations that different components pick from inconsistently; (2) every discovery/matching call site (Reconciler's `ContainerList` filter, Recovery's equivalent, Rollout's `docker ps --filter`) including a project-scope check, most naturally Docker Compose's own `com.docker.compose.project` label, which requires no new label emission from the generator at all since Compose already sets it; (3) the mesh-network lookup convention unified to suffix-matching (already proven correct in `rollout.go`) instead of exact-matching, removing the stated reason for the global hardcoded name.

## 12. ADR Requirement Decision

**Required.** This is not a single-function patch because it requires **coordinated agreement across three components that currently disagree**, plus a decision this project's own governance process (per `docs/adr/README.md`'s lifecycle, referenced by ADR-0006's own header) should make deliberately rather than as a bundled side effect of an outage fix: specifically, whether project-scoping becomes a first-class label Orbit emits itself, or whether it's satisfied by reading Compose's already-present standard label (the latter requires zero generator changes and is the evidence-preferred answer per §8/§9, but is a decision, not an implementation detail). The ADR must also explicitly amend or supersede ADR-0006's specific claim that instance-label scoping "becomes unnecessary" for Reconciler — that claim is being falsified by this review for the cross-project case and should not be silently patched around without updating the record that made it.

## 13. Smallest Safe Implementation Boundary (for future implementation — not performed now)

Everything identified fits within existing package boundaries; no new packages, no changed ownership. In dependency order: (1) generator emits no new labels — Compose's own `com.docker.compose.project` is already present on every container without any change; (2) `Reconciler`'s `ContainerList` filter and `extractBackend` gain a project-scope check and the constructor gains whatever parameter is needed to know its own expected project — mirroring `DockerRecoverySource`'s existing pattern, not inventing a new one; (3) `Rollout`'s three `docker ps --filter`/`docker compose` call sites gain the same project scope; (4) `Reconciler.extractBackend` gains a `VerifyBackendByID`-equivalent short-circuit (or the two are unified into one shared helper) so a live rollout's dynamic ID is recognized rather than collided against; (5) `recovery.go`'s and `reconciler.go`'s exact-match mesh-network lookups are rewritten to suffix-match like `rollout.go`'s `pickMeshIP`, only after (2)-(4) land, since project-scoping is the actual isolation fix and the network-name change is contingent on it, not a substitute for it.

## 14. Tests Required Before Implementation Can Be Accepted

- A live, two-Compose-project test with intentionally identical service names, run concurrently, asserting zero cross-project registration/traffic at the Reconciler layer (extending the pre-market audit's own live reproduction into a permanent regression test).
- A rollout-in-progress test asserting the Reconciler never removes a backend whose dynamic ID the Registry currently holds, across at least the number of reconciliation passes a realistic stability window would span.
- A test constructing two containers with identical static `ORBIT_BACKEND_ID` values (the exact collision condition) and asserting `Reconciler` resolves it correctly rather than logging-and-guessing.
- A unit test for the mesh-network lookup change confirming suffix-matching behaves identically to the existing exact-match behavior for the un-prefixed case, and additionally succeeds for a project-prefixed name (regression-proofing `rollout.go`'s already-correct behavior against the other two components' upcoming change).
- A test asserting `docker orbit rollout`'s internal Compose invocations only ever discover containers within the invoking project, using two same-named services in different projects live.

## 15. Remaining Unknowns

- Whether `com.docker.compose.project` is reliably present for every supported Compose invocation style this project claims to support (explicit `-p`, `.env`-derived, directory-basename-derived) — **unverified**, should be confirmed live before relying on it as the scoping mechanism.
- Whether any currently-deployed (pre-this-ADR) Orbit installation depends on the current cross-project-visible behavior in some unintentional way that a scoping fix would break — **unverified**, no evidence found either way, but per `CONSTITUTION.md`'s migration notes elsewhere in this project, no installed base is claimed to exist yet, which would make this a non-issue if still true.
- Whether shared-proxy mode's `ORBIT_PROXY_INSTANCE` (currently just the alphabetically-first service's name, per `generator.go`'s `buildSharedProxyService`) needs to become a genuinely process-scoped identifier as part of this same ADR, or whether it can remain as-is once project-scoping is added elsewhere — **unverified, flagged for the ADR to decide explicitly rather than resolve implicitly.**
