# ADR-0007 Final Acceptance Review — Proxy Self-Identity Bootstrap

**Date:** 2026-07-15
**Scope:** Resolve the one blocker flagged against ADR-0007: where does "this proxy's own project identity" come from? No code was modified.

---

## 1. Executive Decision

**Option B — runtime, Docker-derived self-identity — is correct and live-verified. Option A is structurally unreliable, not merely more complex, and is rejected.** ADR-0007 is amended (§9 below details exactly what changed) and, with that amendment applied, is recommended for **ACCEPT**.

A second, unanticipated finding surfaced during this review and required a real correction to ADR-0007's original §10, not just an addition: `ORBIT_PROXY_INSTANCE` is not merely an ownership-comparison value today — `internal/config/services_config.go:133` and multiple call sites in `cmd/docker-orbit/main.go` (399, 409, 444, 455, and others) use `cfg.ProxyInstance` as **the load-bearing default service name for legacy single-service proxies**. ADR-0007's original decision to redefine this env var to mean "project name" would have silently broken that synthesis path. This review corrects that: `ORBIT_PROXY_INSTANCE` keeps its existing meaning and role, untouched; project identity becomes a new, separate, runtime-only value that is never sourced from an env var at all.

## 2. Generator-Resolution Analysis (Option A)

**Can `docker orbit generate` resolve the exact project name Docker Compose will later use, without reimplementing Compose's resolution semantics? No — not even in principle, regardless of implementation effort.**

**[code-proven]** `generateCmd` (`cmd/docker-orbit/main.go:84-86`) has exactly three flags today — `--file`, `--output`, `--shared-proxy` — no `--project`/`-p`, no `.env` parsing, no `COMPOSE_PROJECT_NAME` awareness anywhere in `internal/compose`. `ComposeFile` (`internal/compose/models.go:38-43`) has no `Name` field and no catch-all `RawFields` for unrecognized top-level keys — a top-level `name:` in a user's own input file would currently be silently dropped on regeneration, a small, separate, pre-existing bug worth noting but not the focus here.

Even granting a hypothetical future `generate` that fully replicates Compose's precedence rules (explicit `-p` > `COMPOSE_PROJECT_NAME` env > `.env`'s `COMPOSE_PROJECT_NAME` > in-file `name:` > directory basename) and writes a `name:` key into the generated file: **this only pins the *lowest*-precedence fallback.** Nothing prevents an operator from later running `docker compose -p <different-name> -f docker-rollout-compose.yml up -d`, which Compose will honor over any in-file `name:`. `generate` and `up` are separate, decoupled invocations — often run by different tooling, at different times, in different shells with different environments (a CI generate step and a separate deploy step being the ordinary case, not an edge case). **Every place a generator-resolved value could disagree with Docker: any explicit `-p` at `up`-time; any `COMPOSE_PROJECT_NAME` set in the deploy environment but not the generate environment (or vice versa); any `.env` file present at `up`-time but not at `generate`-time (or edited between the two); a compose file moved to a different directory between generate and deploy.** None of these are exotic — CI pipelines routinely generate once and deploy under environment-specific project names. Option A can be *usually* right; it cannot be *guaranteed* right, and the ownership invariant this whole ADR exists to establish requires a guarantee, not a good guess.

## 3. Runtime Docker-Derived Identity Analysis (Option B)

**Can the proxy deterministically locate its own container and read its own `com.docker.compose.project` label? Yes — live-verified across every required scenario.**

Mechanism: Docker assigns every container a default hostname equal to its own short (12-character) container ID, unless a compose service explicitly sets `hostname:` (`internal/compose/generator.go` never does this for the proxy service it injects — **[code-proven]**, confirmed by grep, zero matches). A process inside the container can read this via the standard library (`os.Hostname()` in Go — no shell, no external binary, works even in the proxy's `FROM scratch` image with no `hostname` binary installed, which is why the live tests below used `docker inspect --format '{{.Config.Hostname}}'` rather than `docker exec ... hostname`). That string is then a valid, unambiguous container-ID prefix for a normal `ContainerInspect` call — the exact same Docker client the proxy already holds and already uses for every other discovery operation.

**[live-proven]** — Docker 28.1.1, Compose 2.36.2, real generated Orbit proxy images, this session:

| Scenario | Result |
|---|---|
| Bare container, default hostname | Hostname (`b0522eae6df5`) == short container ID exactly |
| Real generated legacy proxy (`docker orbit generate`, no flags) | Proxy's `Config.Hostname` (`5708a42bc605`) → `docker inspect` on that value resolves to the proxy's own container, `com.docker.compose.project=selfid-proj` returned correctly |
| Proxy container restart (`docker restart`) | Same container, same ID/hostname, resolves identically after restart |
| Proxy container recreation (`--force-recreate`) | New container, new hostname (`847ec06f0a6d`) — self-lookup re-derives from scratch at the new startup and still correctly resolves `project=selfid-proj`; no stale value carried forward, no process-local memory depended on |
| Shared-proxy mode (2 services, 1 proxy) | Proxy's hostname resolves correctly to `project=selfid-shared`, `orbit.io/services=api,web` visible on the same lookup |
| Two independent Orbit proxies running simultaneously on one host | Proxy 1's hostname (`847ec06f0a6d`) resolves only to itself (`selfid-proj`); proxy 2's hostname (`5fab880f5eb0`) resolves only to itself (`selfid-shared`) — hostnames are host-unique by construction (they equal Docker's own host-unique container IDs), so no ambiguity is possible under normal operation |

**[unverified, explicitly flagged, not assumed]** A custom `hostname:` override on the proxy's own generated service definition was not tested because the generator never sets one — this is a precondition Orbit fully controls (the proxy is Orbit's own injected service, never user-supplied), not an external dependency to hope holds. The ADR amendment (§9) states this as a hard requirement: the generator must never set a custom hostname on the proxy service it injects. Also unverified: behavior under a remote Docker context (`DOCKER_HOST` pointing at a different daemon than the one running the proxy container itself) — flagged as a residual open question (§9), not a blocker, since Orbit's own generated deployments always run the proxy against the local daemon via the mounted socket.

**Cgroup-based self-identification (the classic `/proc/self/cgroup` technique) was checked and rejected as the primary mechanism**: this host runs cgroup v2 (`0::/`, confirmed live), under which the pre-container-ID-embedding cgroup path format this classic technique depends on is not reliably present without additional configuration (`--cgroupns=host`, not the default). Hostname-based lookup has no such dependency and is simpler.

## 4. Bootstrap Trust-Root Analysis

1. **How does the proxy identify itself?** By its own container's Docker-assigned hostname, read via the standard library, no external tooling required.
2. **How does it obtain its own project label?** By calling `ContainerInspect` (the Docker client it already holds) on that hostname value and reading `Config.Labels["com.docker.compose.project"]` from the response.
3. **Why is that source authoritative?** Because it is not a prediction of what Compose will do — it is the literal value Compose already wrote onto the container, at the moment the container was actually created, using whichever `-p`/env/`.env` resolution was actually in effect at that exact moment. There is no other source that can be more current or more correct than reading Docker's own record of its own decision.
4. **What happens if self-identification is ambiguous?** Structurally, it cannot be, under normal operation — Docker container IDs (hence default hostnames) are host-unique by construction, live-confirmed with two simultaneous proxies. The only way to introduce ambiguity is a custom hostname override on the proxy's own service definition, which Orbit fully controls and must never do (§9).
5. **What happens if the project label is missing?** Fail closed at startup (§9) — this must never be silently treated as "no project scope" (which would be exactly the vulnerability this ADR exists to close), and per §5's ten-scenario live verification, the label is reliably present for every Compose invocation style tested; its absence would indicate either a non-Compose container-creation path (out of Orbit's supported surface) or genuine Docker/Compose misbehavior worth failing loudly for.
6. **What happens if Docker is unavailable during startup?** Follows the existing, already-established retry-then-degrade pattern this codebase already uses elsewhere (`executeRecovery`'s bounded retry within `cfg.StartupTimeout`, `config.LoadProxyConfig`'s fail-fast state-directory check) — self-identification should be attempted within that same startup budget, and its failure should be treated with the same severity as any other Docker-unavailable-at-startup condition, not a new, separate failure mode.
7. **Can an environment variable override Docker-observed identity?** No. This is the central correction from ADR-0007's original draft (§9).
8. **Can generated configuration disagree with Docker?** Only for the parts of generated configuration that remain generator-resolved (none, for project identity, after this amendment) — the project identity itself is never part of generated configuration at all, so there is nothing for it to disagree with.
9. **Which source wins if they disagree?** Not applicable post-amendment — there is exactly one source (Docker, via self-inspection), not two to arbitrate between.

## 5. Security Review

**Can a malicious or misconfigured container cause an Orbit proxy to believe it belongs to another project? No, not through any Docker-label or env-var spoofing path, and this review found no other path either.**

- **Label spoofing**: not possible for the proxy's *own* identity — Docker labels are fixed at container-creation time and are not writable at runtime by any other process, including another container with Docker socket access (creating/replacing a container with different labels requires exactly the same privilege level already required to run `docker compose up` for that project in the first place — this is not a new trust boundary, it is the same one this project's `SECURITY.md` already documents: "Docker socket access... equivalent to root on the host"). A different, unrelated container cannot reach in and alter the proxy's own already-created container's labels.
- **Environment-variable spoofing**: this is the actual reason to prefer Option B over keeping any env-var-based project assertion authoritative — env vars are materially easier to end up wrong by accident (a hand-edited compose file, a secrets-manager override, a copy-pasted `.env`) than a value Docker itself assigned at creation time from the real, actually-used invocation. This review explicitly recommends **against** introducing any environment variable as an authoritative or even automatically-populated source for project identity, precisely to avoid opening an easier misconfiguration/spoofing path than the one being closed.
- **Container-name collision**: Compose container names are already project-prefixed by construction (`<project>-<service>-<n>`) — a name collision across projects is already prevented by Compose itself, independent of anything this ADR changes.
- **Hostname assumptions**: the one precondition this mechanism depends on (default, unoverridden hostname) is fully within Orbit's own control for its own generated proxy service, not an assumption about arbitrary external containers.
- **Docker socket trust**: unchanged — this mechanism uses the same already-mounted, already-documented, already-accepted Docker socket the proxy already relies on for every other operation. It does not expand that trust boundary; it uses it to answer one more question.

**No cryptographic identity is warranted or proposed** — Docker itself is already the accepted trust root for this class of process, exactly as the review brief anticipated, and inventing a second, parallel trust mechanism (signed tokens, mutual TLS between proxy and generator, etc.) would be disproportionate complexity solving a problem Docker's own existing guarantees already solve.

## 6. `ORBIT_PROXY_INSTANCE` Decision

**Neither of the four options as originally framed fits cleanly, because this review found the env var already has a legitimate, load-bearing, unrelated responsibility that must not be disturbed. The precise decision:**

`ORBIT_PROXY_INSTANCE` **keeps its current meaning and role entirely unchanged**: the default/legacy service name a single-service proxy fronts, consumed by `internal/config/services_config.go:133`'s single-service synthesis fallback and by the several `cfg.ProxyInstance`-keyed call sites in `cmd/docker-orbit/main.go` (399, 409, 444, 455, and others) that wire `ProjectRegistry`, the control server's default-service resolution, and startup-result lookups. **[code-proven]** This is closest in spirit to the review's own framing of Option C ("informational... never used for ownership decisions") but stated precisely: it was never really an ownership mechanism except incidentally, via `recovery.go`'s comparison — and that one incidental use is exactly what's being replaced. ADR-0007's original §10, which proposed redefining this env var to mean "project name," is **withdrawn** — implementing it as originally drafted would have broken the legacy single-service synthesis path this env var already correctly serves. Project identity becomes an entirely separate, new, runtime-only value (not an env var, not generator-emitted, never appearing in a generated compose file) populated exclusively via §3's self-inspection mechanism.

## 7. `orbit.io/proxy-instance` Decision

**D — remove.** This label (`internal/compose/generator.go:304`, set on backing containers to the service's own name) has always been exactly redundant with `orbit.io/service` (set to the identical value, same function, same containers) — **[code-proven]**, confirmed by re-reading `buildBackingService`. Its one consumer, `recovery.go`'s ownership check (`recovery.go:239,246-251,341`), is being replaced by the project+service ownership tuple ADR-0007 §9 already defines, which reads `orbit.io/service` directly and no longer needs a second, duplicate label carrying the same value under a different key. Removing it costs nothing (no other consumer exists, confirmed by exhaustive grep) and removes one more piece of the "each component invented its own partial identity concept" problem this whole ADR exists to close.

## 8. Fail-Closed Startup Semantics

Formalizing §4's answers as explicit, checkable startup behavior: the proxy must perform self-inspection (§3) during startup, within the same bounded retry budget already used for other Docker-unavailable-at-startup conditions (`cfg.StartupTimeout`, the existing pattern in `executeRecovery`). If self-inspection succeeds and returns a non-empty `com.docker.compose.project` value, that value becomes this process's authoritative project identity for every subsequent ownership check (§9's tuple), for the life of the process. If Docker is unavailable for the entire retry budget, or if self-inspection succeeds but the returned label is empty, **the proxy must refuse to start** — silently proceeding with no project scope would recreate exactly the vulnerability this ADR exists to close, and this project's own established convention (state-directory writability, fail-fast, no silent degraded-but-running mode for a load-bearing precondition) already sets this precedent elsewhere in the codebase.

## 9. ADR Sections Changed

ADR-0007 is amended as follows (apply directly to `docs/adr/ADR-0007-backend-identity-and-ownership.md`):

- **§7 Identity Hierarchy** — the "Proxy process" row's "Source" column changes from "`ORBIT_PROXY_INSTANCE` env, resolved at proxy startup" to "self-inspection: the proxy's own Docker-assigned hostname, resolved via `ContainerInspect` against its own container at startup — never an env var."
- **§8 Backend Identity Contract** — no change to the canonical ID shape (`<project>-<service>-<container-id>`); clarify that the `<project>` component is populated from self-inspection, not from any generated or environment-supplied value.
- **§9 Ownership Contract** — the ownership tuple's "project match" check now reads: "matches the value this proxy process discovered about itself via self-inspection at startup (§3 of the acceptance review) — never a value read from `ORBIT_PROXY_INSTANCE` or any other environment variable."
- **§10 Proxy Instance Semantics — substantially rewritten.** Original text redefining `ORBIT_PROXY_INSTANCE` to mean "project identity" is withdrawn in full. Replacement: `ORBIT_PROXY_INSTANCE` retains its original, pre-ADR-0007 meaning (default/legacy service name) unchanged; project identity is a new, separate, self-discovered, non-environment-variable concept, introduced fresh by this amendment, not layered onto an existing field.
- **§12 Docker Source-of-Truth Amendment** — strengthened, not changed in substance: the formal invariant now explicitly names self-inspection as the required source for a proxy's own project identity, closing what would otherwise be an unstated gap in "derive identity/ownership from the same Docker-observable facts" (a proxy's belief about *itself* is exactly such a fact, and must be sourced the same way).
- **§14 Component Responsibility Matrix** — no row changes; the "ownership information required" column already specified "`com.docker.compose.project` == own project" for `Reconciler`/`DockerRecoverySource`/`Rollout` — this amendment specifies precisely where "own project" comes from, without altering any row's substance.
- **§15 Failure Semantics** — add one new row: self-inspection fails or returns an empty project label at startup → **fail startup entirely** (not skip, not degrade) — distinct from the existing per-container skip/degrade rows, because this is a precondition for every other check in that table to mean anything, not a per-container discovery outcome.
- **§16 Migration and Compatibility** — the `ORBIT_PROXY_INSTANCE` migration-impact paragraph is withdrawn (no behavior change to this env var after all); add: no generated-artifact change is required for project identity at all, since it is never generator-emitted.
- **§22 Test Strategy** — add: a test asserting self-inspection returns the correct project value across restart and recreation (already live-verified manually in §3 above; needs a permanent automated equivalent); a test asserting startup fails closed when self-inspection cannot resolve a project (simulate via a container with an overridden hostname, or a mocked Docker client returning an empty label).
- **§23 Live Verification Requirements** — add self-inspection correctness (restart, recreation, shared-proxy, two-simultaneous-proxies) to the required live-verification checklist alongside the existing cross-project traffic tests.
- **§27 Open Questions** — the first open question ("whether the generator should resolve... or whether the proxy should instead read its own... label back at startup") is now **resolved** by this review (the latter) and should be removed from Open Questions and folded into the accepted decision. Add one new, narrower open question in its place: behavior under a remote Docker context (`DOCKER_HOST` pointing elsewhere) is unverified and should be closed before relying on this mechanism in that specific configuration, if Orbit ever supports it.
- **Final Decision Answer 5** ("What does proxy instance mean?") — changes from "the Compose project this proxy process was started for" to: "`ORBIT_PROXY_INSTANCE` continues to mean the default/legacy service name it always meant; the Compose project a proxy belongs to is a separate, self-discovered fact with no environment-variable representation at all."

## 10. Final ADR Acceptance Recommendation

## ACCEPT ADR-0007 (as amended above)

The proxy's own project identity has exactly one authoritative source — self-inspection via its own Docker-assigned hostname, live-verified across restart, recreation, shared-proxy mode, and simultaneous multi-proxy operation — and fail-closed bootstrap semantics are now explicit (§8 above). The correction to §10 (keeping `ORBIT_PROXY_INSTANCE`'s existing meaning rather than repurposing it) removes a real risk of implementation breakage that would have recreated a version of the exact disagreement this ADR exists to eliminate, this time over the proxy's own identity rather than a backend's. With this amendment applied, ADR-0007 is ready for implementation planning as small, PR-scoped, independently reviewable stages, per its own §24.
