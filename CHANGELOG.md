All notable changes to Orbit are documented here.

Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). Versioning follows [Semantic Versioning](https://semver.org/) once the first tag is cut — see `docs/governance/RELEASES.md` for the policy.

## [Unreleased]

### Fixed — Cold-start reliability (Phase 3.2)

A live redeploy of a real 6-service Compose stack — the most basic possible workflow, `docker orbit generate` then `docker compose up`, no rollout ever run — surfaced nine real defects across two sessions, all concentrated in the same place: the per-service proxy's discovery/recovery path. See [ADR-0006](docs/adr/ADR-0006-shared-proxy-and-event-driven-discovery.md) for the architecture decision this exercise produced.

**First-deploy backend registration (five bugs, one session):**
- Generated proxy services never mounted `/var/run/docker.sock` — the proxy couldn't reach the Docker daemon at all.
- Generated backing services were missing the `orbit.io/generation`/`orbit.io/proxy` labels and `ORBIT_BACKEND_ID` env `internal/proxy.DockerRecoverySource.extractBackend` requires — every container looked "incomplete" and was skipped.
- The `docker_rollout_mesh` network wasn't pinned with `name:`, so Compose project-prefixed it and recovery's exact-match network lookup never found a container's IP.
- Proxies had no per-service instance scoping (`ORBIT_PROXY_INSTANCE` / `orbit.io/proxy-instance`), so every proxy discovered every other service's backends too, and could adopt one as its own authoritative generation.
- `internal/state.GenerateRecoveryPlan`'s backend-selection phase filtered candidates by the pre-inference `authority` variable instead of `plan.AuthoritativeGeneration`, so the inferred-fallback cold-start path (no persisted state yet) always produced zero restorable candidates even after correctly inferring an authority.

**Startup and recovery reliability (four bugs, second session):**
- Startup recovery gave up after a single failed TCP dial instead of using its own `ORBIT_STARTUP_TIMEOUT` budget; any backend slower than one dial timeout to open its port (Grafana, cadvisor) read as permanently `failed`. Fixed with a bounded retry loop covering both `StartupFailed` and `StartupRecovering` (a backend with its own Docker `HEALTHCHECK` reporting "starting" was just as stuck).
- The same one-shot-no-retry gap existed one level up, in connecting to the Docker daemon itself — extended the same retry budget to cover daemon reconnects.
- `ProxyConfig.StateDir` (`/var/lib/orbit`) was never mounted as a volume, so `ActiveGenerationState`/`RolloutState` were lost on every container recreation. Fixed with one named volume per proxied service. **Found in the process:** `WriteActiveGenerationState`/`WriteRolloutState` are only ever called from chaos test scenarios, never from any live code path — the volume mount makes persistence durable *if* something writes to it, but nothing does yet. Tracked as a follow-up; not fixed in this pass because doing so incompletely (persisting only at boot, not also when a live rollout hands off authority) would introduce a stale-authority regression worse than today's always-infer behavior.
- `internal/proxy.HealthController` only re-checks backends already in its registry (by its own documented Layer 5 contract, it must never touch Docker lifecycle) — a service that never got past initial recovery, or one whose only backend later crashed, had nothing left to bring it back except a human running `docker orbit recover`. Added a periodic rediscovery pass, gated to only run when a service has zero active backends. Verified live: a proxy stuck at `proxy_status: failed` self-healed to `ready` within one tick of its backend coming back, no CLI command run.

### Removed — `internal/stack` duplicate execution layer (ADR-0005 stage 2)

Not a deletion of the package — [ADR-0005](docs/adr/ADR-0005-multi-service-orchestration-architecture.md) explicitly freezes `internal/stack` as a future v2 subsystem and its dependency-graph engine is real, needed, and unchanged. This executed three of the four items already listed in that ADR's own migration plan: removed `docker_integration.go` (a duplicate deployment engine whose every method was an unconditional placeholder — e.g. `CreateContainer` returned a fabricated `mock-<name>-<timestamp>` ID without touching Docker), `docker_client.go`'s `RealDockerClient` and all of `docker_sdk_client.go` (same placeholder pattern, SDK-flavored — `MockDockerClient` and the `DockerClient` interface are unchanged and still back `docker_transaction.go`/`health_monitor.go`), and `state_persistence.go` (a duplicate WAL nothing else referenced). `internal/stack`: 4,149 → 2,822 non-test LOC, and it no longer imports `github.com/docker/docker` at all, narrowing `docs/governance/SECURITY.md`'s `docker@v24` CVE note.

### Fixed — `docker orbit` display-text consistency and install UX

The CLI's own `--help` output, and several error hints, still said `docker-orbit` (the binary name) in places, while `deploy`/`doctor`/`history`/`status` already correctly said `docker orbit` (how it's actually invoked as a Docker CLI plugin). Root command now uses cobra's `CommandDisplayNameAnnotation` so `--help` renders correctly without changing what Cobra dispatches on internally; `docs/cli-reference/` generation strips the annotation so those file names stay on their existing hyphenated convention. Separately, `make install-plugin` no longer re-runs the full build under `sudo` — only the final copy escalates, and only when the target directory isn't writable, so a plain `go build`'s module cache stays owned by the invoking user instead of root.

### Added — Native distribution & release pipeline (Phase 3.1)

Orbit can now be installed as a native Docker CLI plugin without Go, a source checkout, or a container wrapper. Full guide: [docs/installation.md](docs/installation.md).

- **GoReleaser pipeline** (`.goreleaser.yaml`): cross-compiled binaries for Linux + macOS × amd64 + arm64, `tar.gz` archives, a SHA256 `checksums.txt`, and native **deb/rpm** packages (via nfpm). Reproducible builds (`CGO_ENABLED=0`, commit-pinned `mod_timestamp`), version injected into the existing `main.version`. Validated locally with `goreleaser release --snapshot --clean` (no GitHub Release required). A commented `signs:` block documents the code-signing extension point.
- **Native installer** (`install.sh`, full rewrite): detects OS/arch, resolves the release archive, **verifies its SHA256 checksum and refuses to install on mismatch**, installs into the Docker CLI plugins dir (system, with per-user fallback), and upgrades in place without touching runtime state. `ORBIT_DIST_DIR=./dist ./install.sh` installs from local snapshot artifacts so the whole flow is testable before anything is published. The previous **container-wrapper installer is retired** — it ran the CLI inside an ephemeral container, silently losing the host-side rollback/history state Orbit depends on.
- **deb/rpm packages**: binary at `/usr/bin/docker-orbit` + symlink into `/usr/local/lib/docker/cli-plugins/`. No package-owned config, so uninstall is non-destructive (never removes deployment state).
- **Release workflow** (`.github/workflows/release.yml`): tag-triggered (`v*`), runs build/vet/test/lint then GoReleaser; guarded to the canonical repo.
- **Makefile**: added `dist`, `dist-check`, `install-local`, `clean-dist`; retired the container-wrapper `docker-install` target. Dev workflows unchanged.

**Real bug found and fixed during release validation** — `docker orbit <cmd>` was broken whenever Docker forwarded any global flag. Docker invokes a plugin as `docker-orbit [docker global flags…] orbit <args>` (e.g. `docker --context prod orbit deploy web`), injecting its own flags *before* the plugin-name token. `internal/plugin.StripPluginArgs` stripped only `argv[1] == "orbit"`, so any global flag left `orbit` in place and Cobra rejected it as an unknown command. This was never caught because prior testing exercised the standalone binary or plain `docker orbit x` with no flags. Fixed: plugin mode is now detected via the `DOCKER_CLI_PLUGIN_ORIGINAL_CLI_COMMAND` env var Docker sets (not `argv[0]` — the standalone binary shares the name and may legitimately pass "orbit" as a service argument), and everything up to and including the plugin-name token is stripped. Verified end-to-end through real Docker (`docker --config <dir> orbit version/doctor/--help`); regression tests added in `internal/plugin/plugin_test.go` (the package's first tests).

### Fixed — Production reliability hardening (Phase 3.0)

Reliability-focused phase — no new commands or strategies. Systematic crash-recovery validation, failure injection, concurrency stress, soak, performance baselines, security audit, chaos, and observability review. Full report: [docs/reliability-report.md](docs/reliability-report.md). Three real defects were found in **shipped** code by the new tests and fixed:

- **`internal/state.IsTransitionStale` was not progress-aware.** It measured staleness only from `TransitionStart`, ignoring `LastProgressAt` (whose own field doc says it "resets timeout clock"). A slow-but-healthy drain that made steady progress but exceeded the overall timeout was wrongly flagged stale → recovery went `degraded`. Now: an absolute deadline is the hard cap, otherwise staleness is measured from the last observed progress; a transition that keeps progressing is not stale, one that stalls past the timeout is. Fail-safe direction either way (over-conservative before, accurate now). This resolves the long-standing `TestIsTransitionStaleWithProgress` failure.
- **`internal/state.AtomicWriteJSON` was not concurrent-safe.** It wrote to a fixed `path+".tmp"`, so two concurrent writers to the same path collided — one writer's `rename` removed the temp the other was about to open, producing a spurious "no such file" error (the chaos `StateManagerConcurrency` scenario reproduced this). Now each writer uses a unique `os.CreateTemp` file (0600, fsync, atomic rename — last-writer-wins, never a torn file); a stale legacy `path+".tmp"` is still cleaned up on write for crash recovery. Production callers already serialize per-service via advisory + in-process locks, so this is defense-in-depth for the primitive — but it closes a real corruption-adjacent hazard in the state layer.
- **`internal/rollout.AcquireLock` reported a false "corrupted" error under concurrency.** The `O_EXCL` create and metadata write are two steps; a concurrent acquirer that read the lock file in between saw it empty and returned `lock file corrupted — manual inspection required` instead of the clean "already in progress". A new 16-way contention stress test surfaced this. Fixed with a bounded read-retry (`readLockMetadataWithRetry`) that distinguishes the microsecond create-vs-write window from genuine corruption. Mutual exclusion was always correct (exactly one winner); only the loser's error message was wrong.

**New reliability tests** (all `-race`-clean): crash-recovery scenario + 50×-determinism validation (`internal/state/recovery_scenarios_test.go`), graceful failure-injection at each rollout step (`internal/rollout/failure_injection_test.go`), 16-way lock-contention stress (`internal/rollout/lock_concurrency_test.go`), rate-limiter goroutine-leak detection (`internal/api/leak_test.go`), and state-file 0600 permission regression guards (`internal/state/security_test.go`). The whole `internal/state` package (state persistence + recovery — success criterion "never corrupts deployment state") is now green under `-race`, as is the full 25-scenario chaos suite.

**Security finding (not yet changed — needs product sign-off):** the control API binds all interfaces (`:port`) and authentication is opt-in (`ORBIT_API_TOKEN` unset ⇒ mutating `/backends` and `/recover` are served unauthenticated, rate-limiting being the only default protection). The token mechanism is enforced correctly when set. Secure-by-default (bind localhost and/or require a token) is the primary item gating a PRODUCTION READY rating — see the report's Security Review.

### Added — Production deployment workflow (Phase 2.2)

Three commands delivering Orbit's primary capability — safe production deployments — all built on the existing `internal/rollout` engine established in Phase 2.0. No new orchestration logic, no experimental deployment strategies, no redesign of recovery.

- **`docker orbit deploy <service>`** — the production front end over the same `rollout.Run` engine `docker orbit rollout` calls, adding: pre-flight safety checks (the identical checks `doctor` runs — Docker reachable, compose file valid, proxy reachable *and* healthy, recovery state consistent), a plan preview showing current generation/backend health/the exact step sequence that will run, a confirmation prompt (skippable with `--yes` for CI), live per-phase progress reporting (`pulling`, `scaling_up`, `health_check`, `registering`, `draining`, `deregistering`, `complete` — via a new `rollout.Options.Progress` callback invoked at each real step transition, not synthesized), and a completion summary. `docker orbit rollout` remains unchanged for scripts already depending on its exact behavior. Stable exit codes distinguish outcomes: `0` success, `1` deploy ran but failed, `2` aborted before any change (bad compose file/service, or `--json` without `--yes`), `3` pre-flight failed, `4` succeeded with unhealthy backends remaining.
- **`docker orbit rollback <service>`** — promoted from an unstructured command to the same preview/confirmation/JSON/progress treatment as `deploy`, over the same `rollout.Rollback` engine (now instrumented with a `RollbackProgressFunc` mirroring `deploy`'s `Progress`, and recording a `history.EventRollback` entry on every outcome — success or failure — via a `defer`, so a crash mid-rollback still leaves a record). `--to <generation>` is honestly scoped to what the engine's state model actually supports: exactly one prior generation is ever persisted, cleared once a deployment completes, so `--to` can only *confirm* that one recorded target — attempting anything else returns a clear error naming the one target that is recoverable, rather than pretending a multi-generation history exists.
- **`docker orbit recover`** — an on-demand trigger for the identical recovery pass the proxy already runs at startup, now exposed via a new `POST /recover` control API endpoint (`internal/api/recovery.go`). `cmd/docker-orbit/main.go`'s startup recovery logic was refactored into a shared `executeRecovery` function called from both the startup path and `POST /recover` via `ControlServer.SetRecoveryTrigger` — one implementation, two call sites, so the CLI can never diverge from what actually happens when the proxy boots. Recovery pass is serialized (409 if one is already in-flight) and deterministic: if authority can't be established, it reports that plainly (`action: "degraded"`) and exits non-zero rather than guessing, and distinguishes a fully-confirmed restore from an `inferred_fallback` (recovered, but with no persisted authority to confirm against — shown as a warning, not silently treated as full success).

**Real bug found and fixed via live testing**: `internal/api.RateLimiter` used a token-bucket burst of 1 regardless of the configured rate. `doctor` (4 sequential control-API calls), `deploy` (pre-flight + plan + result), and `recover` (status + recover) all issue several rapid sequential requests from the same client IP per invocation — burst-of-1 meant a single legitimate CLI invocation could 429 itself. Fixed by setting burst equal to the configured rate limit (the standard `golang.org/x/time/rate` idiom): sustained abuse is still capped at the same rate, but one command's own request sequence is no longer treated as multiple abusive clients. Caught by actually running the proxy binary and driving `deploy`/`recover` against it end-to-end, not by unit tests alone — regression-guarded by `TestBurstAllowsRapidSequentialRequestsFromOneCommand`.

**Also fixed**: `deploy`'s pre-flight-failure path called `os.Exit(ExitUnavailable)` only in human-output mode — in `--json` mode it printed the plan (correctly showing `"preflight_passed": false`) but returned exit code `0`, silently breaking the "stable exit codes" guarantee for scripted/CI callers relying on `--json`. Both branches now exit identically; regression-guarded by a subprocess-based test (`TestDeploy_UnreachableProxy_ExitsUnavailable` in the new `exitcode_test.go`).

**Testing**: 57 new tests this phase (269 → 326), including realistic end-to-end flows rather than isolated function tests wherever the engine allows it — `internal/rollout/rollback_test.go` drives `Rollback` against a real `httptest` server implementing the exact `/backends` register/drain/deregister routes (no mocks), covering the interrupted-deployment scenario (old backend still recorded, new one draining), 409-tolerance on re-registration, and history recording on both success and failure. `cmd/docker-orbit/recover_test.go` covers the full status-code contract of `POST /recover` (200/409/401/403/503/500) and the three distinct outcome renderings (success/`inferred_fallback`/`degraded`) so the "never guess" behavior is asserted, not just described. A new `exitcode_test.go` finally implements the subprocess-based exit-code testing pattern `deploy_test.go` referenced by name since Phase 2.1's gap-fill pass but never created — `os.Exit` paths in `renderCLIErr` can't be tested in-process without killing the test binary, so these re-exec the test binary itself and assert on the real subprocess exit code.

**Manually verified end-to-end against a live proxy** (not just unit tests): registered a real backend, ran `deploy --dry-run` (plan preview with live pre-flight results), `recover` (correctly reported `degraded` — honest, since the registered backend had no corresponding persisted rollout state or real container for recovery to independently confirm against — demonstrating the never-guess guarantee), and a full `rollback` cycle (restore → drain 2s → deregister → complete), then confirmed `docker orbit history` recorded the real rollback event.

### Added — Operational CLI foundation (Phase 2.1)

Three new commands, each answering one of Orbit's Product Philosophy questions ("what is happening", "what happened", "is everything healthy"), all calling the real, existing engine — no mocked behavior, no duplicated state:

- **`docker orbit status`** — live deployment state: current/previous generation, deployment phase, proxy status, healthy/unhealthy backends (checked via a real TCP probe at request time), active traffic target, and recovery counters. Supports `--json`, `--watch` (continuous polling), `--project` (verifies the queried proxy's reported service name). Backed by a new `GET /status` endpoint on the control API (`internal/api/status.go`).
- **`docker orbit history`** — recorded rollout/rollback timeline for a service, newest first. Backed by a new `internal/history` package (append-only JSONL event log) wired into `internal/rollout.Run` and `Rollback` — history accumulates from when this shipped forward; there is no retroactive record, and the command says so explicitly rather than implying data loss.
- **`docker orbit doctor`** — 10 real diagnostic checks (Docker Engine connectivity, Docker Compose v2 CLI availability, compose file validity, control-address/config validity, proxy reachability, proxy readiness, required-port availability, recovery-state consistency, state-directory writability, plugin installation), each classified PASS/WARNING/ERROR with a required remediation step for anything not PASS. Never displays a raw stack trace.

**New shared infrastructure**, reusable by every future command:
- `internal/cli/output` — consistent human/JSON rendering and standardized exit codes (`ExitOK`=0, `ExitError`=1, `ExitConfig`=2, `ExitUnavailable`=3, `ExitDegraded`=4)
- `internal/cli/clierr` — standardized CLI errors: every error requires a `What`/`Why`/`Action`, enforced at construction time (a missing remediation step panics rather than shipping)

**Closed a real, pre-existing gap found during Phase 1.95's tech debt audit**: `internal/api.DebugHandler` existed with real logic (recovery/authority/rollout metrics) but was never instantiated or wired to any HTTP route, and its `Record*` methods were never called — meaning even if wired up, it would have reported empty data. Both gaps are now closed: `DebugHandler` is instantiated in `cmd/docker-orbit/main.go`'s `runProxy`, wired to `ControlServer` via `SetDebugHandler`, and its `RecordActiveGenState`/`RecordRolloutState`/`RecordRecoveryPlan` are called at the real points those values are computed during the recovery flow. `internal/metrics.MetricsCollector`'s recovery-timing methods (`RecordRecoveryStart`/`RecordRecoveryFailure`/`RecordAuthorityTransition`/`SetCurrentState`) are now called too — verified end-to-end with a live local proxy run (`recovery_count` incremented from a real recovery pass, not a hardcoded value).

**Auto-generated CLI documentation**: a hidden `docker-orbit docs` command (`make docs`) regenerates `docs/cli-reference/` from the live Cobra command tree via `cobra/doc` — the reference can't drift from actual `--help` output because it's generated from the same source.

**New documentation**: `docs/troubleshooting.md`, mapping real error conditions to the `doctor` checks that catch them.

**Test coverage**: `cmd/docker-orbit` had zero test files before this phase (flagged in Phase 2.0's consolidation report); it now has real coverage for `status`, `history`, and `doctor`'s logic, including golden-file tests for all three commands' human-readable output (`cmd/docker-orbit/testdata/*_golden.txt`, regenerable via `UPDATE_GOLDEN=1 go test ./cmd/docker-orbit/...`). `internal/cli/output`, `internal/cli/clierr`, `internal/history`, and `internal/api/status.go` are all new and fully covered, including live-socket tests (real TCP listeners, real `httptest` servers) rather than mocks.

**Phase 2.1 gap-fill pass**: audited the three commands against Phase 2.1's original requirements and closed what was actually missing, without re-architecting anything already working:
- `doctor`: added "Docker Compose available" (real `docker compose version` invocation — distinct from validating a specific compose *file*, which the pre-existing "Compose file" check already covered) and "Required ports available" (parses the compose file's declared host ports via a newly-exported `compose.ParsePort` and probes each with a real `net.Listen`, reusing the exact parsing logic `docker orbit generate` uses rather than reimplementing it). A bound port reports WARNING, not ERROR — it's the expected state once a service is already deployed, and this check can't distinguish "Orbit's own proxy" from "something else" from the host side alone.
- `cmd/docker-orbit`: added `history_test.go` — `historyCmd`'s own rendering and JSON-shape logic had no test file (only the underlying `internal/history` package was covered).
- Deliberately **not** implemented: a "recovery" entry in `docker orbit history`'s timeline. Recovery executes inside the proxy container; `internal/history` writes to the CLI host's filesystem. Recording recovery events from inside the container would silently write to a path `docker orbit history` can never read on the host — see `docs/troubleshooting.md`'s "recovery event" entry for the full reasoning and the real signal (`docker orbit status`'s `Recovery` fields, `docker orbit doctor`'s "Recovery state consistent" check) that already covers this without a half-working timeline entry.
- Also considered and deliberately not implemented: a "Configuration version" doctor check (no versioned config file exists yet — Orbit's configuration is entirely environment-variable driven) and a distinct "State integrity" check beyond what "Recovery state consistent" already surfaces (the proxy's persisted state lives inside the container, not somewhere the host-side CLI can inspect directly). Both would have required inventing something to check rather than exposing something real, which Phase 2.1 explicitly rules out.

### Removed — Docker CLI plugin architecture consolidation (Phase 2.0)

- Deleted the entire `plugin/` directory (Docker managed-plugin implementation: `plugin/cmd/docker-orbit/`, `plugin/proxy/`, `plugin/config.json`, and associated Dockerfile/Makefile/docs). Direct code inspection found it had **no working deployment logic** — every command handler printed hardcoded progress text or returned hardcoded JSON (e.g. `isProxyRunning()` was a literal `return false // Placeholder`; the "proxy" binary had no TCP proxying code at all). It imported zero `internal/` packages and had zero test coverage. Full evidence and reasoning in [ADR-0002's Resolution section](docs/adr/ADR-0002-docker-cli-plugin-architecture.md#resolution-phase-20).
- Deleted `docs/hybrid-mode.md` and `docs/plugin-integration.md` — both documented only the removed mock architecture, not Orbit's real design.
- **Canonical Docker CLI plugin architecture going forward**: `cmd/docker-orbit/` + `internal/plugin/` — the standalone-binary-plus-`argv[0]`-detection pattern matching `docker compose`/`docker buildx`/`docker scout`, calling the real, tested engine (`internal/compose`, `internal/rollout`, `internal/proxy`, `internal/state`, `internal/api`).

See `BRAND.md` for the full specification and rationale.
- Resolved four inconsistent, pre-existing organization/registry names found across the repository (`mdumair`, `antigravity-labs`, `antiersolutions`, `technicaltalk`) to a single canonical identity.

### CI/CD

- Split the test suite into a fast, deterministic blocking gate (`internal/api`, `compose`, `config`, `metrics`, `plugin`, `proxy`, `rollout`, `volumes`, `testing/concurrency`, `testing/profile` — passes cleanly in ~28s) and a non-blocking "known issues" step (`internal/stack`, `internal/state`). ### Known Issues (tracked, not yet fixed)

- `internal/rollout`: `TestAcquireLockSuccess`, `TestAcquireLockBlocked`, `TestGetProcessStartTicks`, `TestIsProcessAlive`, `TestLockRelease`, `TestLockMetadataPreserved` fail on macOS — `lock.go`'s process-liveness check reads `/proc/[pid]/stat`, which is Linux-only (already documented as such in the code's own comments). These pass on Linux (including CI, which runs on `ubuntu-latest`) and are a local macOS-dev-only artifact, not a product defect; a real cross-platform fix (e.g. `kill(pid, 0)` + a platform-appropriate start-time source) is Phase 4 (Production Hardening) scope, not addressed here.

### Resolved Known Issues

- `internal/stack`: the data races in the mock Docker client / transaction test harness (`TestEmitEvent`, `TestTransactionFullRollout`) are fixed — `TestEmitEvent` now synchronizes on a channel instead of a racy bool+sleep, and `MockDockerClient` now guards its maps with a mutex (plus a `SetContainerHealth` helper for tests that flip health from a background goroutine). Package is `-race`-clean. Note: `internal/stack` is still experimental and unwired from the CLI (see `internal/stack/README.md`) — this only means its own tests no longer destabilize the blocking gate, not that the package is used in v1 deployments.
- `internal/state`: `TestIsTransitionStaleWithProgress` now passes — the progress-aware staleness fix already recorded above under "Fixed" resolved it; this entry just removes it from Known Issues to match. Package is `-race`-clean.
